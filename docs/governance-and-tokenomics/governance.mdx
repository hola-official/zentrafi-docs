## Governance Model

ZentraFi uses vote-escrowed governance: a battle-tested model where staking duration and amount determine voting power.

The model creates alignment between governance participants and protocol success:

- Long-term stakers have more influence
- Active participation is rewarded
- Short-term speculation is discouraged

## The veZTR System

### Overview

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Stake ZTR  │ ──► │ Accrue veZTR │ ──► │    Vote     │
│             │     │             │     │             │
│  Earn sZTR  │     │  Max: 1000x │     │ Direct      │
│  (revenue)  │     │  over 7 days│     │ emissions   │
└─────────────┘     └─────────────┘     └─────────────┘
```

### Staking Mechanism

When you stake ZTR:

| Outcome | Description |
|---------|-------------|
| **sZTR** | Share of protocol trading fees |
| **veZTR** | Voting power for governance |

Both accrue simultaneously from a single staking action.

**Key Properties:**
- No deposit/withdrawal fees
- No lock period required
- Stake/unstake anytime
- Unified rewards (both sZTR and veZTR from one stake)

### sZTR: Revenue Sharing

sZTR represents your share of protocol revenue:

**Revenue Sources:**
- Trading fees from Zentra X
- Launch fees from Zentra App
- Protocol fees (where applicable)

**Distribution:**
- Fees collected continuously
- Converted to ZTR
- Distributed proportionally to stakers

**Calculation:**
$$
\text{Your sZTR Rewards} = \frac{\text{Your Staked ZTR}}{\text{Total Staked ZTR}} \times \text{Protocol Fees}
$$

<Info>
sZTR rewards represent real yield from protocol activity, not token emissions.
</Info>

### veZTR: Voting Power

veZTR is non-transferable voting power that accrues to ZTR stakers.

**Accrual Mechanics:**

$$
\text{veZTR}(t) = \min\left(\text{ZTR} \times \frac{t}{7 \text{ days}}, \text{ZTR} \times 1000\right)
$$

| Property | Value |
|----------|-------|
| Maximum | 1000× staked ZTR |
| Time to maximum | 7 days |
| Accrual | Linear |
| Transferable | No |

**Critical:** Unstaking any amount resets your entire veZTR balance to zero.

### Adding More Stake

When you add to your stake:

1. Pending sZTR rewards are claimed automatically
2. Your veZTR maximum increases
3. New stake begins accruing veZTR from zero
4. Existing veZTR is preserved

## Gauge System

Gauges control how ZTR emissions are distributed across liquidity pools.

### How Gauges Work

```
veZTR Holders
     │
     ▼
Vote for Pools
     │
     ▼
Gauge Controller
     │
     ▼
Calculate Weights
     │
     ▼
Distribute Emissions
     │
     ▼
LPs Earn Rewards
```

### Voting Power Formula

Pool weight uses a power model:

$$
w_i = v_i^{\alpha}
$$

Where:
- $w_i$ = Weight for pool $i$
- $v_i$ = Votes for pool $i$
- $\alpha = 2/3$

**Why 2/3 power?**

This model encourages distribution rather than concentration:
- Diminishing returns on large vote allocations
- Incentivizes spreading votes across pools
- Prevents single pools from dominating emissions

### Emission Distribution

$$
\text{Pool Emission} = \frac{w_i}{\sum w} \times \text{Total Emissions}
$$

### Voting Rules

| Rule | Detail |
|------|--------|
| Vote anytime | Changes take effect immediately |
| Split votes | Allocate across multiple pools |
| Redirect | Withdraw and reassign freely |
| Before unstake | Must remove all votes first |

## Voting Rewards

A permissionless incentive layer where anyone can reward voters for specific allocations.

### Mechanism

1. **Creator** deposits reward tokens for a specific pool
2. **veZTR holders** vote for that pool
3. **Voters opt-in** to the reward program
4. **Rewards distribute** proportionally to votes

### Why This Exists

Projects want liquidity in their pools. They can:
- Buy votes directly through voting rewards
- Compete for emissions without governance control
- Create predictable incentive costs

### Participation

```
1. Vote for pool
2. Navigate to Voting Rewards
3. Find reward programs for your voted pools
4. Opt-in to program
5. Claim rewards periodically
```

<Warning>
Voting alone doesn't earn rewards. You must explicitly opt-in to Voting Reward programs.
</Warning>

## Governance Security

### Timelock

All governance actions pass through a timelock:

- Minimum delay before execution
- Community review period
- Cancellation possible during delay

### Thresholds

| Action Type | Requirement |
|-------------|-------------|
| Standard proposals | Quorum + majority |
| Parameter changes | Higher quorum |
| Emergency actions | Multi-sig + timelock |

### Constraints

Governance cannot:
- Modify immutable core contracts
- Bypass timelock (except defined emergencies)
- Access user funds directly
