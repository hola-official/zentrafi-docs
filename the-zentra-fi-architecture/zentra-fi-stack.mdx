## Architecture Overview

ZentraFi is structured as a modular protocol stack. Each layer has distinct responsibilities and can evolve independently while maintaining interoperability with adjacent layers.

```
┌─────────────────────────────────────────────────────────────────┐
│                      APPLICATION LAYER                          │
│    Zentra App  •  Zentra X Interface  •  Analytics Dashboard    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                       PROTOCOL LAYER                            │
│      Launchpad Contracts  •  Liquidity Book  •  Router          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      GOVERNANCE LAYER                           │
│    veZTR Voting  •  Gauge Controller  •  Treasury  •  Timelock  │
└─────────────────────────────────────────────────────────────────┘
```

## Application Layer

The application layer provides user-facing interfaces for interacting with protocol contracts.

### Zentra App (Launchpad Interface)

**Purpose:** Token creation, launch participation, and graduation tracking.

**Components:**
- Token launch wizard with metadata configuration
- Buy/sell interface for bonding curve trading
- Real-time price and graduation progress
- Post-graduation trading transition

### Zentra X Interface (DEX)

**Purpose:** Trading, liquidity provision, and position management.

**Components:**
- Swap interface with route optimization
- Liquidity management dashboard
- Position analytics and P&L tracking
- Pool explorer and market data

### Analytics Dashboard

**Purpose:** Protocol-wide metrics and performance tracking.

**Components:**
- Volume and TVL tracking
- Fee generation analytics
- Launch performance metrics
- Governance participation stats

## Protocol Layer

The protocol layer contains the core smart contracts that implement ZentraFi's functionality.

### Launchpad Contracts

The launchpad uses a modular architecture for token launches with bonding curve mechanics.

```
┌─────────────────────────────────────────────────────────────────┐
│                        LaunchpadBonding                         │
│  (Main entry point - launches tokens, manages graduation)       │
└──────────────────────────┬──────────────────────────────────────┘
                           │
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ LaunchpadFactory│ │  LaunchpadRouter │ │  LaunchpadPair  │
│ (Creates tokens)│ │ (Handles swaps) │ │   (AMM pool)    │
└────────┬────────┘ └─────────────────┘ └─────────────────┘
         │
         ▼
┌─────────────────┐
│ LaunchpadToken  │
│  (ERC-20 token) │
└─────────────────┘
```

**LaunchpadBonding**: Main entry point for launching tokens and managing graduation.

```solidity
interface ILaunchpadBonding {
    function launch(
        LaunchParams calldata params
    ) external payable returns (address token, address pair);

    function buyWithNative(
        address token,
        uint256 minTokens,
        uint256 deadline
    ) external payable returns (uint256 tokensReceived);

    function sellForNative(
        address token,
        uint256 amount,
        uint256 minNative,
        uint256 deadline
    ) external returns (uint256 nativeReceived);

    function isEligibleForGraduation(
        address token
    ) external view returns (bool eligible, uint256 remaining, uint256 threshold);
}
```

**Key Functions:**
- `launch`: Deploy new token with bonding curve pool
- `buyWithNative`: Purchase tokens with native currency
- `sellForNative`: Sell tokens for native currency
- Graduation triggers automatically when token reserve < 20%

### Liquidity Book Contracts

**LBFactory**: Registry and factory for liquidity pools.

```solidity
interface ILBFactory {
    function createLBPair(
        IERC20 tokenX,
        IERC20 tokenY,
        uint24 activeId,
        uint16 binStep
    ) external returns (ILBPair pair);
}
```

**LBPair**: Individual liquidity pool implementing bin-based trading.

```solidity
interface ILBPair {
    function swap(
        bool swapForY,
        address to
    ) external returns (bytes32 amountsOut);

    function mint(
        address to,
        bytes32[] calldata liquidityConfigs,
        address refundTo
    ) external returns (
        bytes32 amountsReceived,
        bytes32 amountsLeft,
        uint256[] memory liquidityMinted
    );

    function burn(
        address from,
        address to,
        uint256[] calldata ids,
        uint256[] calldata amountsToBurn
    ) external returns (bytes32[] memory amounts);
}
```

**LBRouter**: Aggregation and routing for optimal execution.

```solidity
interface ILBRouter {
    function swapExactTokensForTokens(
        uint256 amountIn,
        uint256 amountOutMin,
        Path memory path,
        address to,
        uint256 deadline
    ) external returns (uint256 amountOut);

    function addLiquidity(
        LiquidityParameters calldata liquidityParameters
    ) external returns (
        uint256 amountXAdded,
        uint256 amountYAdded,
        uint256 amountXLeft,
        uint256 amountYLeft,
        uint256[] memory depositIds,
        uint256[] memory liquidityMinted
    );
}
```

## Governance Layer

The governance layer manages protocol parameters and resource allocation.

### veZTR (Vote-Escrowed ZTR)

**Purpose:** Governance token representing voting power.

**Mechanics:**
- Users stake ZTR to receive veZTR
- veZTR accrues linearly over 7 days to maximum (1000x stake)
- Unstaking resets veZTR balance to zero
- veZTR is non-transferable

### Gauge Controller

**Purpose:** Manages emission allocation across pools.

**Functions:**
- Records veZTR votes for pools
- Calculates emission weights using power model
- Distributes ZTR rewards proportionally

### Treasury

**Purpose:** Holds protocol-owned assets.

**Management:**
- Controlled by governance proposals
- Funds ecosystem development
- Manages protocol-owned liquidity

### Timelock

**Purpose:** Security mechanism for governance actions.

**Parameters:**
- Minimum delay before execution
- Cancellation capabilities
- Emergency actions for critical situations

## Security Model

### Contract Security

- Audits in progress
- Bug bounty program planned
- Formal verification on critical paths

### Access Control

- Factory contracts: Admin-controlled presets
- Pair contracts: Permissionless after creation
- Governance: Timelock-protected

### Upgrade Path

- LaunchpadBonding, Factory, Router: UUPS Upgradeable
- LBPair contracts: Immutable after creation
- Parameters: Adjustable within bounds via governance
