## Component Overview

ZentraFi consists of four core components, each serving a distinct function in the protocol's operation.

| Component | Function | Layer |
|-----------|----------|-------|
| Launchpad | Token distribution | Protocol |
| Liquidity Book | Trading and LP | Protocol |
| Governance | Parameter control | Governance |
| Treasury | Asset management | Governance |

## Launchpad

The launchpad enables projects to launch ERC-20 tokens with bonding curve mechanics, graduating to TraderJoe's LiquidityBook concentrated liquidity.

### Contract Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        LaunchpadBonding                         │
│  (Main entry point - launches tokens, manages graduation)       │
└──────────────────────────┬──────────────────────────────────────┘
                           │
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ LaunchpadFactory│ │  LaunchpadRouter │ │  LaunchpadPair  │
│ (Creates tokens)│ │ (Handles swaps) │ │   (AMM pool)    │
└────────┬────────┘ └─────────────────┘ └─────────────────┘
         │
         ▼
┌─────────────────┐
│ LaunchpadToken  │
│  (ERC-20 token) │
└─────────────────┘
```

### Core Contracts

| Contract | Type | Purpose |
|----------|------|---------|
| `LaunchpadBonding` | UUPS Upgradeable | Main entry point, graduation logic |
| `LaunchpadFactory` | UUPS Upgradeable | Token deployment with metadata |

| `LaunchpadToken` | Standard | ERC-20 with transfer restrictions |
| `LaunchpadMath` | Library | AMM and fee calculations |

### Virtual Liquidity AMM

The launchpad uses a Uniswap V2-style constant product formula with virtual liquidity:

$$
x \times y = k
$$

Virtual quote reserves are calculated from two parameters:

$$
\text{Virtual Quote} = \frac{K \times \text{PRECISION}}{\text{assetRate} \times 10^9}
$$

| Parameter | Default | Description |
|-----------|---------|-------------|
| `K` | 100,000,000,000 | Base curve constant |
| `assetRate` | 5,000,000,000 | Adjustable for quote price changes |

### Anti-Sniper Protection

Time-based decaying tax on early trades:

| Time | Tax Rate |
|------|----------|
| 0s | 94% |
| 15s | 47% |
| 30s | 0% |

### Fee Structure

| Fee | Rate | Split |
|-----|------|-------|
| Buy | 1% | 20% creator / 80% treasury |
| Sell | 2% | 20% creator / 80% treasury |
| Launch | 0.01 PROS | 100% treasury |

### Graduation Flow

When token reserve drops below 20%:

1. Liquidity removed from `LaunchpadPair`
2. Token becomes fully transferable
3. sZTR holders receive 10% of raised WPROS
4. Remaining liquidity migrates to LiquidityBook
5. Trading continues on Zentra X

## Liquidity Book

The Liquidity Book is ZentraFi's core trading infrastructure: a discretized AMM that organizes liquidity into price bins.

### Why Discretization?

Traditional constant-product AMMs use continuous curves:

$$
x \cdot y = k
$$

This spreads liquidity across infinite prices. A token at $1 has liquidity at $0.001 and $1000: completely useless capital.

Liquidity Book discretizes into bins:

$$
P_i \cdot x_i + y_i = L_i
$$

Each bin $i$ holds liquidity $L_i$ at a single price $P_i$. Capital only exists where it can be used.

### Bin Mechanics

**Bin Step:**
The price increment between consecutive bins.

$$
P_{i+1} = P_i \cdot (1 + s)
$$

Where $s$ is the bin step (e.g., 0.25% = 25 basis points).

**Active Bin:**
The bin currently containing both token reserves. Only the active bin earns fees.

**Bin Composition:**
Each bin contains reserves following:

$$
c = \frac{y}{L}
$$

Where $c$ is the composition factor (percentage of bin in token Y).

### Trading Flow

1. User submits swap request
2. Router finds optimal path through bins
3. Active bin provides liquidity at fixed price
4. If depleted, price moves to next bin
5. Fees collected and distributed to bin LPs

### Fee Structure

$$
\text{Fee} = \text{Base Fee} + \text{Variable Fee}
$$

**Base Fee:**
$$
f_b = B \cdot s
$$

Where $B$ is the base factor and $s$ is bin step.

**Variable Fee:**
$$
f_v = A \cdot (v_a)^2
$$

Where $A$ is the variable fee control and $v_a$ is the volatility accumulator.

The volatility accumulator increases with:
- Bin crossings during swaps
- Swap frequency
- Large trade sizes

## Governance

Governance coordinates protocol resources through vote-escrowed token mechanics.

### veZTR System

```
Stake ZTR → Accrue veZTR → Vote on Gauges → Direct Emissions
```

**Staking:**
- No lock period required
- Stake/unstake anytime
- Unified staking (sZTR + veZTR simultaneously)

**veZTR Accrual:**
$$
\text{veZTR}(t) = \min\left(\text{ZTR} \cdot \frac{t}{T}, \text{ZTR} \cdot M\right)
$$

Where:
- $t$ = Time since stake
- $T$ = Time to maximum (7 days)
- $M$ = Maximum multiplier (1000x)

**Voting Power:**
Pool weight uses power model:

$$
w_i = v_i^{\alpha}
$$

Where $\alpha = 2/3$ encourages distribution across pools rather than concentration.

### Gauge System

Gauges measure votes and distribute emissions:

1. veZTR holders vote for pools
2. Gauge controller records weights
3. Emissions distribute proportionally
4. LPs claim rewards from farms

### Voting Rewards

Permissionless incentive layer:
- Anyone can create reward pools
- Deposited tokens reward voters
- veZTR holders opt-in to programs
- Rewards proportional to votes

## Treasury

The treasury manages protocol-owned assets through governance.

### Revenue Sources

| Source | Flow |
|--------|------|
| Protocol fees | % of trading fees |
| Launch fees | % of raise |
| Penalty fees | Early unstake, etc. |

### Expenditure Categories

- Liquidity mining emissions
- Development funding
- Ecosystem grants
- Protocol-owned liquidity

### Governance Control

All treasury actions require:
1. Governance proposal
2. veZTR voting period
3. Timelock delay
4. Execution

Emergency actions have separate paths with higher thresholds.
