## What Makes a Launch "Fair"?

A truly fair launch requires:

1. **Equal access**: Anyone can participate under the same rules
2. **Transparent pricing**: No hidden information advantages
3. **Bot resistance**: Automated trading provides no unfair edge
4. **Immediate utility**: Tokens gain full utility upon graduation

ZentraFi's launchpad enforces these principles through smart contract mechanics.

## Anti-Sniper Protection

The launchpad implements **time-based anti-sniper tax** that heavily penalizes early automated trades while rewarding patient participants.

### How It Works

A decaying tax applies to trades immediately after launch:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `antiSniperEnabled` | true | Toggle protection |
| `antiSniperDuration` | 30 seconds | Tax decay window |
| `MAX_ANTI_SNIPER_TAX_BPS` | 9400 (94%) | Starting tax rate |

### Tax Decay

The tax decays **linearly** from maximum to zero:

| Time After Launch | Tax Rate |
|-------------------|----------|
| 0 seconds | 94% |
| 7.5 seconds | 70.5% |
| 15 seconds | 47% |
| 22.5 seconds | 23.5% |
| 30 seconds | 0% |

### Economic Impact

At launch with maximum anti-sniper tax (94%) plus buy fee (1%):

```
User pays: 100 WPROS
Reaches pool: 5 WPROS
Goes to fees/treasury: 95 WPROS
```

After anti-sniper period (just 1% buy fee):

```
User pays: 100 WPROS
Reaches pool: 99 WPROS
Goes to fees: 1 WPROS
```

<Warning>
The total fee is capped at 95% (`MAX_TOTAL_FEE_BPS`), ensuring users always receive some value even during the anti-sniper window.
</Warning>

### Why This Works

| Attack Type | How Anti-Sniper Defeats It |
|-------------|---------------------------|
| **MEV bots** | 94% tax makes front-running unprofitable |
| **Sniper bots** | Instant buys lose most value to tax |
| **Flash loans** | Tax captures most borrowed capital |
| **Insider timing** | No speed advantage; patience wins |

## Fee Structure

### Trading Fees

| Fee Type | Default | Range |
|----------|---------|-------|
| `buyFeeBps` | 100 (1%) | 0-1000 (0-10%) |
| `sellFeeBps` | 200 (2%) | 0-1000 (0-10%) |
| `MAX_FEE_BPS` | 1000 (10%) | Maximum allowed |

Higher sell fees discourage quick flips and reward holders.

### Fee Distribution

Trading fees are split between token creators and the protocol treasury:

| Recipient | Default Share | Parameter |
|-----------|---------------|-----------|
| **Creator** | 20% | `creatorFeeBps = 2000` |
| **Treasury** | 80% | Remainder |

**Example:**
- User sells tokens, paying 2% fee
- Creator receives 0.4% (20% of 2%)
- Treasury receives 1.6% (80% of 2%)

Fees emit the `FeesDistributed` event for transparent tracking.

### Launch Fee

| Parameter | Default | Description |
|-----------|---------|-------------|
| `launchFee` | 0.01 PROS | Fee to create a new token |

This prevents spam launches while remaining accessible.

## Transfer Restrictions

Tokens are **non-transferable until graduation**:

### Allowed Actions Before Graduation

| Action | Allowed | Reason |
|--------|---------|--------|
| Buy from pool | Yes | Via router/bonding |
| Sell to pool | Yes | Via router/bonding |
| Transfer to wallet | No | Prevents off-market trading |
| Transfer to exchange | No | Prevents pre-listing |
| Add to LP | No | No external liquidity |

### Whitelisted Addresses

Only these contracts can receive tokens:
- `LaunchpadBonding` - Main entry point
- `LaunchpadRouter` - Swap execution
- `LaunchpadPair` - AMM pool

Users can transfer TO these addresses (selling), but cannot receive FROM them directly (must go through swap functions).

### Post-Graduation

Once graduated:
- Token becomes fully ERC-20 compliant
- All transfer restrictions removed
- Can list on external exchanges
- Can add to external LPs

## Sybil Resistance

Multiple wallets provide **no advantages**:

| Strategy | Result |
|----------|--------|
| Split purchases | Same total price impact |
| Multiple wallets | No per-wallet caps to exploit |
| Coordinated buying | Still pays curve price |

The constant product AMM treats total demand the same regardless of source.

## Whale Protection

Large purchases face natural resistance:

### Price Impact

The AMM formula creates increasing cost per token:

$$
\text{Tokens Out} = \frac{\text{Token Reserve} \times \text{Amount In}}{\text{Quote Reserve} + \text{Amount In}}
$$

Larger purchases get progressively worse rates.

### Public Execution

All trades are on-chain:
- Large buys are visible to everyone
- No hidden order books
- Community can see accumulation patterns

## Graduation Transition

### The Problem

Most launches fail at transition:

```
Launch Complete → ??? → Trading Active
```

This gap enables manipulation, extraction, and confusion.

### ZentraFi's Solution

Atomic graduation with no gap:

```
Token Reserve < 20% → Graduation Triggers
    │
    ├── 1. Liquidity removed from pair
    ├── 2. Transfer restrictions lifted
    ├── 3. sZTR allocation distributed
    ├── 4. LiquidityBook pool created
    └── 5. Concentrated liquidity seeded
```

All steps execute in sequence. No window for manipulation.

## Evaluating Launch Fairness

### Green Flags

- Anti-sniper protection enabled
- Reasonable fee structure (1-2%)
- 20%+ graduation threshold
- Transfer restrictions active
- sZTR allocation enabled

### Red Flags

- Anti-sniper disabled
- Excessive fees (>5%)
- Very high graduation threshold (>50%)
- Modified transfer restrictions
- No sZTR allocation

### Questions to Ask

1. Is anti-sniper protection enabled and for how long?
2. What are the buy/sell fee rates?
3. How much goes to creator vs treasury?
4. What's the graduation threshold?
5. When do transfer restrictions lift?
